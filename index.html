Runtime Security Controls

Runtime security controls establish the hardened execution environment for live validator nodes. These controls address four threat surfaces: process-level isolation, network perimeter segmentation, key material protection across the full data lifecycle, and binary integrity verification at startup. Controls in this section apply to both Operator Layer (Layer 2) infrastructure and Middleware Layer (Layer 3) coordination nodes.

1.1 Process Isolation Model

Each validator process runs in a dedicated execution namespace with no shared memory regions between the signing daemon, telemetry agent, and network listener processes. The process model enforces a least-privilege execution boundary: the signing daemon holds the narrowest possible permission set, limited to receiving signing envelopes from the designated middleware socket and returning signed blobs to the same socket. It has no outbound network access, no filesystem write access outside a designated ephemeral scratch directory, and no inter-process communication channels other than the controlled middleware socket.

Container-level isolation (e.g., Linux namespaces, cgroup constraints) enforces memory and CPU ceilings per process. A breach of the ceiling threshold triggers an automated alert to the operator SOC before any process termination. Privileged container capabilities are explicitly dropped at startup. Root access to the validator host is prohibited during normal operation; all administrative access requires a signed maintenance session envelope delivered through the same OCP-controlled pathway used for configuration updates.

1.2 Network Segmentation Rules and Firewall Policies

Validator hosts operate within a dedicated network segment, isolated from shared operator infrastructure by a hardware firewall enforcing allowlist-only ingress and egress. No validator host has reachability to the public internet except through explicitly enumerated egress rules that route exclusively to OCP endpoints and the Post Fiat peer network.

Firewall rules are version-controlled and distributed only through the OCP configuration update pathway, which requires a signing ceremony for any rule modification. Operators cannot modify firewall rules through direct host access during normal operations. All rule changes are logged with timestamp, initiating entity, and ceremony nonce to the immutable audit log.

1.3 Key Material Protection: At Rest and In Transit

Key material protection follows the zero-custody boundary established in M2. The Institution (Layer 1) holds all private key material in approved custody: institution-owned HSM (FIPS 140-2 Level 3), air-gapped signing system, or institutional multisig with the institution as policy owner. Cloud HSMs are rejected because the provider retains operational access. This boundary is inviolable: no key material enters Operator Layer (Layer 2) or Middleware Layer (Layer 3) infrastructure under any operational scenario, including incident response and disaster recovery.

Key rotation for operator credentials follows the root-gated signing ceremony pathway defined in M3. A key rotation request requires a Maker/Checker/Reviewer (M/C/R) ceremony inside the institution's custody environment. The Reviewer role is non-waivable for key rotation actions. After rotation, the previous key is revoked in the OCP key registry within 300 seconds. Any signing attempt using a revoked key returns ERR_KEY_REVOKED and triggers a P2 alert.

In-transit protection uses mutual TLS (mTLS) 1.3 for all connections crossing process or host boundaries. Certificate pinning is enforced at both the OCP endpoint and the validator host. A certificate mismatch terminates the connection immediately and generates a P2 alert. Session keys are ephemeral and rotated per connection; no session key is reused across envelope deliveries.

1.4 Secure Boot Verification for Validator Binaries

All validator processes are verified against a cryptographic manifest before execution. The secure boot pipeline runs at host startup and before each process launch following a restart or update event.

1. Manifest Retrieval

OCP delivers a signed binary manifest (SHA-256 hashes + version identifiers) via the OCP bridge channel. Manifest is signed with OCP's Ed25519 root key. Signature is verified before manifest is trusted.

2. Binary Hash Verification

Each validator binary is hashed locally (SHA-256). Local hash is compared against the manifest entry. Any mismatch halts the startup sequence and generates a P1 alert. The affected process does not start.

3. Version Lock Enforcement

Running version is checked against the OCP-enforced minimum version floor. A binary below the minimum version is blocked from starting. The operator receives a remediation window notification through the OCP admin channel.

4. Runtime Integrity Monitoring

A continuous integrity monitor polls each running binary's memory-mapped image every 300 seconds against the trusted manifest hash. Any drift from the expected hash generates a P1 alert and triggers OCP-directed process restart from a verified binary.

Binary Substitution Attack Prevention The 300-second runtime integrity polling window is deliberately short relative to the minimum viable dwell time for a binary substitution attack (estimated 600 seconds for an attacker to establish persistence after substitution). The polling interval prevents an attacker from substituting a binary, performing malicious operations, and restoring the original binary before the next integrity check. The sustain window is one poll. A single hash mismatch triggers the alert immediately with no debounce, because legitimate binary changes arrive only through the OCP update pathway, which pre-registers the new hash in the manifest before the binary is distributed.

Automated Threat Detection and Alerting Pipelines

The automated threat detection pipeline monitors validator node behavior across four threat surface categories: access control, configuration integrity, signing behavior, and network perimeter. Detection logic runs inside the Middleware Layer (Layer 3) telemetry processing pipeline, which is independent of operator-controlled infrastructure. This independence ensures that a compromised operator cannot suppress alerts by interfering with host-level monitoring agents.

2.1 Anomaly Detection Triggers

Each trigger below specifies the detection logic, the sustain window (to prevent false positives from transient conditions), and the resulting alert severity. All triggers route to the operator SOC and the OCP simultaneously. Suppression by either party is not architecturally possible.

2.2 Alert Routing to Operator SOC and OCP

All alerts are routed simultaneously to two independent channels: the operator Security Operations Center (SOC) and the OCP incident queue. Routing is bidirectional and non-exclusive. Suppression at the SOC level does not suppress the OCP alert. The OCP alert channel is operated by the Middleware Layer (Layer 3) and is architecturally independent of Operator Layer infrastructure.

Operator SOC Channels

  • PagerDuty integration (P1/P2 alerts, immediate page)
  • Slack channel #validator-security-alerts (P3/P4 alerts)
  • Email notification digest (P4, hourly batch)
  • SOC dashboard real-time feed (all severities)
  • Webhook to operator SIEM for correlation

OCP Incident Queue Channels

  • OCP incident API endpoint (all severities)
  • OCP automated response pipeline (P1, triggers isolation)
  • Institutional notification gateway (P1/P2)
  • Network coordination alert for P1 events affecting signing
  • OCP audit log (immutable, all severities)

Alert payload format is standardized across all triggers. Each alert record includes: alert identifier (UUID), trigger category, detection timestamp (UTC), validator node identifier, operator identifier, severity classification, sustain window state at trigger time, raw evidence snippet (hash, log line, or connection attempt record), and alert routing confirmation (SOC ACK timestamp required within tier SLA).

Anti-Suppression Design The OCP incident queue endpoint is authenticated using a certificate pinned to the OCP root certificate authority, not the operator's certificate chain. Operator infrastructure cannot intercept, modify, or suppress alert delivery to the OCP queue. If the OCP queue endpoint is unreachable for more than 30 seconds, the validator node enters a safe degraded state (continues block validation; suspends new signing ceremony acceptance) and the operator SOC receives a P1 alert about OCP connectivity loss.

Incident Classification and Escalation Procedures

Incident classification maps the automated alert severity model to a structured human response workflow. The escalation path is designed so that no single party (operator, institution, or OCP) can unilaterally resolve a P1 or P2 incident without cross-layer coordination. This mirrors the governance coordination model established in M3: the signing authority layer and the infrastructure response layer remain separated even under incident conditions.

3.1 Severity Tiers and Response Time SLAs

P1 15 minutes 30 minutes from ACK 4 hours from containment 48 hours; root cause analysis required

SLA Breach Escalation If the operator fails to acknowledge a P1 alert within 15 minutes, OCP escalates automatically: (1) the institution's designated security contact receives a direct notification through the institutional gateway; (2) the OCP network coordination alert is elevated to network-level visibility; (3) the validator node remains in isolated state until operator acknowledgment plus institution concurrence is received. An unacknowledged P1 that persists for 60 minutes without operator response initiates the network-level coordination procedure.

3.2 Escalation Paths: From Automated Alert to Network-Level Coordination

Stage 1: Automated Detection (T+0)

Anomaly detection trigger fires. Alert payload generated and routed simultaneously to operator SOC PagerDuty and OCP incident queue. OCP automated response evaluates severity classification. P1: validator isolation initiated immediately. P2: signing ceremony suspended pending ACK. P3/P4: logged, no automated action.

Stage 2: Operator SOC Response (T+15m for P1)

Operator SOC engineer acknowledges the alert through the OCP incident management interface, not through a direct host action. ACK registers a timestamp, engineer identity, and initial triage assessment. For P1: SOC engineer confirms isolation state and begins forensic evidence preservation sequence. For P2: SOC engineer evaluates whether P2 should be escalated to P1 based on triage.

Stage 3: Institution Notification (P1/P2, T+30m)

OCP delivers a structured incident notification to the institution's security contact through the institutional gateway. Notification includes: incident identifier, severity classification, automated actions taken, operator ACK status, and a non-action request (the institution is being informed, not asked to sign anything at this stage). If the incident involves key material exposure risk, the institution's custody team is notified separately with a key rotation recommendation.

Stage 4: Cross-Layer Coordination (P1 escalation or P2 requiring signing)

If incident resolution requires a root-gated action (key rotation, deregistration, configuration update) the OCP constructs the appropriate signing ceremony envelope and delivers it to the institution through the standard ceremony pathway. The operator has no role in the signing decision. Maker/Checker/Reviewer process inside the institution's custody environment applies with full M/C/R requirements. Reviewer is non-waivable even under incident conditions.

Stage 5: Network-Level Coordination (P1, if operator unresponsive at T+60m)

OCP escalates to network coordination alert. Other active validators in the cohort receive a read-only status notification (no action required from them). OCP evaluates whether the affected validator's allocation weight should be redistributed to remaining operators within the 60% single-operator cap. Redistribution uses the CRS-based routing logic from M3. Network-level coordination does not require institution signing unless a permanent deregistration decision is made.

Stage 6: Resolution and Return-to-Operations (Post-containment)

Resolution is confirmed when: the threat is eliminated and evidence preserved, the validator node has passed secure boot verification with a clean manifest, OCP has confirmed telemetry continuity for 5 consecutive minutes, and the institution has acknowledged the post-incident notification. Operator must submit a root cause analysis within the tier SLA. Return to full signing ceremony operations requires institution sign-off through the OCP gateway. The operator cannot unilaterally restore the node to active signing status after a P1 incident.

Forensic Evidence Preservation Workflows

Forensic evidence preservation ensures that post-incident investigations can reconstruct attacker activity, confirm the scope of compromise, and demonstrate compliance with institutional security standards. Evidence must be preserved in a form that is admissible for institutional audit, regulator review, and potential legal proceedings. All preservation workflows operate independently of operator-controlled infrastructure to prevent evidence tampering by a compromised operator.

4.1 Log Immutability Guarantees

All security-relevant logs generated by validator processes, the telemetry agent, the OCP bridge, and the network listener are written simultaneously to two destinations: a local rolling buffer (readable by the operator SOC) and the OCP immutable log store (write-once, operator-inaccessible for modification or deletion).

Log CategoryLocal RetentionOCP Immutable Store RetentionIntegrity Mechanism
Signing ceremony event logs90 days rolling7 years (regulatory minimum)SHA-256 hash chain; each entry includes hash of previous entry
Authentication and access logs90 days rolling7 yearsSHA-256 hash chain; timestamp from OCP NTP source (not local clock)
Firewall and network egress logs30 days rolling3 yearsSHA-256 hash chain; forwarded within 60 seconds of generation
Process integrity check results30 days rolling3 yearsSHA-256 hash of result record; signed by telemetry agent key
Configuration state snapshots10 versions rollingIndefinite (per version)Content-addressed storage; OCP-signed manifest per snapshot
OCP command and response log90 days rolling7 yearsSHA-256 hash chain; bidirectional; OCP and validator both retain copies

The hash chain mechanism ensures that log tampering is detectable: any modification to an entry invalidates the chain from that entry forward. OCP verifies the hash chain integrity for all logs in the immutable store on a daily basis. A chain integrity failure generates a P2 alert and initiates a forensic investigation of the affected log stream.

Timestamps in all log entries use OCP-sourced NTP time, not the local validator host clock. An attacker with access to the local host cannot alter timestamps by manipulating the local clock. The OCP NTP source is verified against a GPS-disciplined reference and does not accept local network time sources.

4.2 Chain-of-Custody Procedures for Compromised Node Artifacts

When a validator node is isolated following a P1 or P2 incident, a chain-of-custody workflow is initiated immediately. The workflow is triggered automatically by OCP isolation and does not require operator action to begin. This prevents a compromised operator from delaying evidence collection.

Step 1: Isolation Confirmation and Custody Record Creation (T+0 from isolation)

OCP records the isolation event with a UUID custody record identifier. The record includes: trigger alert ID, isolation timestamp, validator node identifier, operator identifier, and the OCP engineer identity (or "AUTOMATED" for automated isolation). This custody record is the root of the chain-of-custody log for all subsequent evidence actions.

Step 2: In-Memory State Capture (T+2m, before any process termination)

The OCP bridge instructs the validator host to capture a memory dump of all validator processes before any process is terminated. The dump is written to an isolated evidence partition with a computed SHA-256 hash. No network traffic is permitted from the host during this step. The hash is registered in the custody record. The evidence partition is mounted read-only after capture.

Step 3: Log Buffer Flush and Forward (T+5m)

All local log buffers are flushed and forwarded to the OCP immutable store before any process restart. The OCP immutable store returns a receipt with a content hash confirming receipt of each log stream. Receipts are added to the custody record. Log forwarding is confirmed before proceeding to Step 4.

Step 4: Disk Image Snapshot (T+10m)

A block-level disk image of the validator host root volume is created and stored in the evidence partition. The image is hashed (SHA-256) and the hash registered in the custody record. The image is uploaded to a designated OCP forensic storage endpoint over a one-way transfer mechanism (the forensic storage endpoint cannot write back to the validator host). The image is encrypted at rest using a key held by OCP, not the operator.

Step 5: Custody Record Finalization and Institutional Notification (T+15m)

The custody record is signed by OCP and delivered to the institution's security contact via the institutional gateway. The operator receives a read-only copy of the custody record summary (not the underlying evidence). The full evidence set is accessible only to OCP forensic analysts and the institution's designated forensic contacts. Operator access to forensic evidence requires a co-signed authorization from OCP and the institution.

Operator Evidence Access Restriction A compromised operator is the primary adversary model for chain-of-custody design. Operators cannot access the OCP forensic storage endpoint. Operators cannot modify or delete entries in the OCP immutable log store. Operators cannot view the full custody record without institution co-authorization. This design ensures that even a fully compromised Operator Layer cannot destroy forensic evidence needed to investigate the compromise.

4.4 Integration with M2 Disaster Recovery Plan

The forensic evidence preservation workflow integrates with the disaster recovery architecture established in M2. The DR plan defined the recovery hierarchy: standby node promotion, key rotation via signing ceremony, and institutional re-enrollment. The forensic workflow connects to this hierarchy at two integration points.

First, forensic evidence collection precedes DR activation. OCP will not promote a standby node to replace a compromised validator until the custody record creation step (Step 2 above) is confirmed. This prevents DR activation from overwriting forensic state on the compromised host before evidence is captured. The 15-minute evidence capture window is designed to complete before any DR activation would be operationally required for a P1 event.

Second, the forensic log store feeds the root cause analysis required before the recovered validator is returned to active signing. The institution reviews the forensic findings as part of its decision to authorize the return-to-operations signing ceremony. No institution is required to authorize re-activation without reviewing the forensic summary. The signing ceremony for re-activation is explicitly linked to the incident record ID in the envelope payload, so the institution knows exactly which incident recovery it is authorizing.

OCP Recovery Integration Points

The Orchestration Control Plane (OCP) serves as the recovery coordination authority for all security incidents affecting validator nodes. OCP recovery integration follows the same layered authorization model used for routine operations: automated actions for time-critical containment, human-in-the-loop authorization for institutional decisions, and root-gated signing ceremonies for permanent state changes. The OCP cannot take permanent recovery actions unilaterally. Institutions retain signing authority over all root-gated operations even during incident response.

5.1 Validator Isolation Triggering

OCP triggers validator isolation through the OCP bridge channel, not through direct host access. The isolation command is authenticated with OCP's mTLS certificate and includes: incident record ID, isolation justification code, authorized action scope, and a freshness timestamp within the 120-second signing window (preventing replay of a stale isolation command to take a validator offline outside an incident context).

Isolation Freshness Protection Isolation commands include a 120-second freshness timestamp using the same mechanism as signing ceremony envelopes. A replayed isolation command (for example, an attacker attempting to take a validator offline by replaying a captured isolation command from a previous incident) is rejected with ERR_NONCE_REPLAY. The 30-day nonce retention log applies to isolation commands as well as signing envelopes. This prevents isolation commands from being weaponized as availability attacks.

5.2 Standby Failover Initiation

Failover to standby nodes follows the weight redistribution logic established in M3 routing specifications. The CRS-based routing engine redistributes the isolated validator's allocation weight across remaining active operators subject to two hard constraints: no single operator may hold more than 60% of total allocation weight, and at least two active operators must remain in the active pool. If redistribution cannot satisfy both constraints (for example, only two operators remain and one is isolated), OCP enters a safe degraded state and escalates immediately to the institution for guidance.

Standby failover weight redistribution is logged in the OCP audit trail with the triggering incident record ID. Institutions can trace any routing change back to the incident that caused it. Failover weight changes are temporary by default: when a previously isolated validator is cleared and returned to operations, its allocation weight is restored through the normal CRS-based routing cycle, not through an emergency override that would bypass the scoring logic.

5.3 Fleet-Wide Security Patch Coordination

A security incident on one validator may indicate a vulnerability applicable to the full validator fleet. OCP fleet-wide patch coordination allows a security fix to be distributed to all operators simultaneously while maintaining the signing ceremony integrity requirements for binary changes.

Stage 1: Patch Authorization

OCP security team prepares a patch manifest with new binary hashes for all affected components. The manifest is signed with OCP's root signing key. The patch is classified as a root-gated action (configuration update class). A signing ceremony envelope is constructed for each institution, referencing the patch manifest hash. Institutions perform the M/C/R process independently before any patch is distributed to operators.

Stage 2: Staged Rollout

After institution authorization, OCP distributes the patch to operators in stages: 25% of operators first, with a 4-hour observation window before proceeding to the full fleet. If any patched validator shows anomalous behavior during the observation window, the rollout is paused and the affected validator is isolated pending investigation. Patch rollout status is visible to all institutions through the OCP governance dashboard.

Stage 3: Operator Distribution

Each operator receives the patch through the OCP update channel. Operators apply the patch through their standard update pipeline. After application, the secure boot verification sequence runs automatically, confirming the new binary hashes match the authorized manifest. Operators report patch completion status to OCP through the telemetry channel. Any operator that fails to apply an emergency security patch within the specified remediation window is downgraded in CRS scoring under the configuration freshness component.

Stage 4: Verification and Fleet Attestation

After full fleet rollout, OCP generates a fleet attestation report confirming that all validators are running the authorized binary versions per their registered manifest hashes. The report is delivered to all institutions through the institutional gateway and retained in the OCP immutable audit store. Institutions can independently verify any operator's binary version by requesting a telemetry attestation through the OCP dashboard.

Emergency Patch Timeline For actively exploited vulnerabilities (incident-linked patches), the staging observation window is compressed to 1 hour from 4 hours. Institution authorization is still required, but OCP may request expedited M/C/R processing with a 2-hour ceremony SLA rather than the standard 24-hour window. Reviewer authorization is non-waivable even under emergency patch conditions. The Reviewer role exists specifically to function as a check during high-pressure situations.

5.4 Restoration of Normal Operations After Incident Resolution

Return to full operations after a P1 or P2 incident follows a defined clearance sequence. No step in this sequence can be skipped, and the institution holds sign-off authority over the final restoration action.

The operator cannot return a validator to active signing operations by self-certifying resolution. This is the defining control that separates the Post Fiat security model from conventional operator-controlled infrastructure: the institution's signing authority governs not only the validator's initial enrollment but also its restoration after every significant security incident. Operators who breach this sequencing are treated as governance violations under the M3 coordination model.
